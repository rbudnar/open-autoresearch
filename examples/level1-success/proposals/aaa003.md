# Experiment Proposal: Attention pooling encoder

## Hypothesis

Because the baseline mean-pooling encoder loses positional information across long input sequences, switching to attention pooling (`attention_pool`) should reduce NLL — particularly on long-sequence subgroups — at the cost of a modest latency increase.

## Literature basis

Offline mode. Reference is a one-line note in canon.bib: attention-pooling encoders are a standard architecture-branch first attempt for sequence inputs. No novel-architecture claims.

## Proposed change

Add `models/encoders/attention_pool.py` implementing a small additive-attention pooling layer. Wire `encoder_type: attention_pool` into the training config. No other config switches change.

## Single non-baseline switch

```yaml
config:
  encoder_type: attention_pool
# all other switches pinned to baseline
```

## Files expected to change

- `models/encoders/attention_pool.py` (new)
- `configs/experiments/aaa003.yaml` (new, derived from baseline)
- `models/encoder_factory.py` (registry entry)

## Protected files that must not change

- `evaluation/**`, `metrics/**`, `data/splits/**`

## Evaluation plan

- **Smoke (Stage A):** 100 steps on 64 examples.
- **Proxy (Stage B):** 5% train split.
- **Full validation (Stage C):** 3 seeds.
- **Ablation type:** Not at Level 1.

## Early stopping rule

Stop if proxy NLL after 1 epoch is > baseline proxy NLL + 0.05.

## Risks

- **Leakage:** None — architectural change only.
- **Compute:** Within max_gpu_hours_per_candidate = 3.
- **Metric gaming:** Attention pooling can over-attend to a single position; subgroup behavior on short sequences should be checked in the report.
- **Implementation:** Single attention head; multi-head sweep deferred to Level 3.
